# Hash

## 개념
키-값을 매핑하는 구조  
내부적으로 해시함수를 이용하여 키를 배열의 인덱스로 변환하고, 해당 위치에 값을 저장한다.   
평균적으로 O(1) 시간에 삽입, 검색, 삭제가 가능하다

## 핵심 구성요소
- 해시함수: 임의 크기의 데이터를 고정된 크기값으로 변환한다. 좋은해시함수는 키를 균일하게 분포시켜 충돌을 최소화한다.
- 버킷 : 해시값에 대응하는 실제 저장공간이다. 배열의 각 슬록이 하나의 버킷이다.
- 충돌: 서로다른키가 같은 해시값을 가지게되는상황이다. 어떤해시함수든 입력 공간이 출력공간보다 크면 충돌이 불가피하다.

## 해시 충돌 해결 방법
1. 체이닝: 같은버킷에 연결리스트(or트리)를 두어 충돌된항목들을 저장한다. 구현이 간단하고 삭제가쉽다.
2. 개방주소법: 충돌시 다른 빈 버킷을 탐색한다.
   1. 선형탐사: 다음칸을 순서대로 탐색, 클러스터링 문제있음
   2. 이차탐사: 제곱간격으로 탐색. 클러스터링을 줄임
   3. 이중해싱: 두번쨰 해시함수로 탐지 간격 결정. 가장 균일한 분포

## 시간복잡도
- 검색 : 평균 O(1) / 최악 O(n)
- 삽입 : 평균 O(1) / 최악 O(n)
- 삭제 : 평균 O(1) / 최악 O(n)

## 면접대비 질문

### 해시 자료구조에 대해 설명해 주세요.
해시테이블은 키를 해시함수에 통과시켜 얻은 해시값을 배열의 인덱스로 사용하여 값을 저장하는 자료구조이다.  
평균적으로 삽입, 조회, 삭제가 O(1)에 동작하는것이 장점이다.  
하지만, 서로다른 키가 같은 해시값을 갖는 충돌이 발생할수있어서 충돌 해결 전략이 반드시 필요하다. 

### 값이 주어졌을 때, 어떻게 하면 충돌이 최대한 적은 해시 함수를 설계할 수 있을까요?
키를 해시 공간에 균등하게 분포시켜야한다.  
입력값의 작은 차이도 해시값에 크게 반영되도록 비트를 잘섞어야하고, 특히 테이블 크기가 2의 거듭제곱인 경우 하위 비트가 치우치지않도록 상위비트를 섞는 과정이 중요하다.  
키의 특성에 맞는 설계도 필요한데, 예를들어 문자열일면 문자 순서를 반영하는 방식으로 누적곱 연산을 사용하는것이 좋다.

### 해시값이 충돌했을 때, 어떤 방식으로 처리할 수 있을까요?
두가지 방법이 있다.  
첫번째는 Separate Chainig(분리연결법)으로 같은버킷에 연결리스트나 트리형태로 저장하는 방식이다.  
두번재는 Open Addressing(개방주소법)으로 충돌이 나면 다른빈슬록을 탐색하는 방식이다.  
체이닝은 구현이 단순하고 삭제가 쉽다. 개방주소법은 캐싱효율이 좋다.


### 본인이 사용하는 언어에서는, 어떤 방식으로 해시 충돌을 처리하나요?
자바의 해시맵은 기본적으로 분리연결법 방식을 사용한다.   
그리고 자바8부터는 한 버킷에 엔트리가 일정 개수 이상 쌓이면 연결 리스트를 트리구조로 변환해 최악 시간 복잡도를 O(n)에서 O(log n)으로 낮추도록 개선되었다. 

### Double Hashing 의 장점과 단점에 대해서 설명하고, 단점을 어떻게 해결할 수 있을지 설명해 주세요.
더블 해싱은 개방주소법에서 충돌발생시 두번째 해시 함수를 사용해 다음 탐색 위치를 결정하는 방식이다.  
장점은 선형탐사에서 발생하는 1차 클러스터링을 줄일수있다.  
단점은 해시 계산 비용이 증가하고, 두번째 해시함수 설계가 잘못되면 특정 슬롯만 반복 탐색할 위험이 있다.  
이를 해결하기위해 두번쨰 해시값이 0이 되지 않도록 보장하고, 테이블 크기와 서로소가 되도록 설계해야한다. 

### Load Factor에 대해 설명해 주세요. 본인이 사용하는 언어에서의 해시 자료구조는 Load Factor에 관련한 정책이 어떻게 구성되어 있나요?
로드 팩터는 현재 저장된 데이터 수를 전체 버킷수로 나눈 값이다.  
값이 높아질수록 충돌 가능성이 증가하고 성능이 저하된다.  
자바 해시맵은 기본적으로 로드팩터가 0.75이며, 메모리 사용과 성능사이의 균형을 맞추기위해 이 임계치를 넘으면 테이블을 2배로 확장하고 재해싱을 수행한다.

### 다른 자료구조와 비교하여, 해시 테이블은 멀티스레드 환경에서 심각한 수준의 Race Condition 문제에 빠질 위험이 있습니다. 성능 감소를 최소화 한 채로 해당 문제를 해결할 수 있는 방법을 설계해 보세요
전역 락을 사용하는 방식은 구현은 단순하지만 성능저하가 크다.  
그래서 락을 분할해 버킷단위로 동시성을 제어하는 방식을 사용한다. 자바의 ConcurrentHashMap이 이방식을 사용한다.  
또한, 읽기 연산은 가능하면 락없이 처리하고, 리사이즈는 점진적으로 수행해 긴 정지시간을 줄이는 방식으로 설계할수있다. 
핵심은 락을 최소단위로 줄이고, 경합이 발생할 확률 자체를 낮추는것이다.
