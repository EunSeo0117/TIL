# 기타 알고리즘

## 면접대비 질문

### Thread Safe 한 자료구조가 있을까요? 없다면, 어떻게 Thread Safe 하게 구성할 수 있을까요?
일부 언어에서는 Thread Safe 한 자료구조를 제공한다.  
예를 들어 Java에서는 ConcurrentHashMap, CopyOnWriteArrayList와 같은 concurrent collection이 있다.  
하지만 대부분의 일반적인 자료구조는 기본적으로 Thread Safe 하지 않다.  
Thread Safe 하게 구성하기 위해서는 여러 가지 방법이 있다.  
가장 일반적인 방법은 synchronized나 ReentrantLock과 같은 lock을 사용해 임계 영역을 보호하는 것이다.  
또한 AtomicInteger와 같은 원자적 연산을 지원하는 클래스를 사용하여 lock 없이 thread-safe 연산을 수행할 수도 있다.  
이외에도 내부적으로 동기화를 처리하는 concurrent 자료구조를 사용하거나 immutable 객체를 사용하여 데이터 변경 자체를 없애는 방법도 있다.

### 배열의 길이를 알고 있다면, 조금 더 빠른 Thread Safe 한 연산을 만들 순 없을까요?
배열의 길이가 고정되어 있다면 전체 자료구조에 하나의 lock을 거는 대신 배열의 각 구간이나 인덱스 별로 lock을 분리하는 방식으로 병렬성을 높일수있다.  
이를 Lock Striping방식이라고 한다.  
이렇게 하면 여러 스레드가 서로다른 위치에 동시 접근 할수 있기때문에 성능이 개선된다.  
또한 AtomicIntegerArray와 같은 atomic배열을 사용하면 CAS기반 연산을 통해 lock없이도 thread-safe연산을 수행할수있다. 

### 주요 프로그래밍 언어의 자료구조는 Thread Safe 한가요? 그렇지 않다면, Thread Safe 한 Wrapped Data Structure 를 제공하고 있나요?
대부분의 프로그래밍 언어의 기본 자료구조는 스레드세이프하지않다.  
이는 불필요한 동기화로 인한 성능 저하를 방지하기 위해서이다.  
대신 언어별로 스레드세이프한 자료구조나 wrapper를 제공한다.  
예를 들어 Java에서는 기본 컬렉션인 ArrayList나 HashMap은 Thread Safe 하지 않지만 Collections.synchronizedList()와 같은 synchronized wrapper를 제공하며, ConcurrentHashMap과 같은 concurrent 컬렉션도 제공한다. Python의 경우 기본 자료구조는 Thread Safe 하지 않지만 queue.Queue와 같은 동시성 자료구조를 제공한다. C++ STL 컨테이너 역시 기본적으로 Thread Safe 하지 않기 때문에 mutex 등을 사용하여 직접 동기화를 구현해야 한다.

### 문자열을 저장하고, 처리하는 주요 자료구조 및 알고리즘 (Trie, KMP, Rabin Karp 등) 에 대해 설명해 주세요.

Trie는 문자열을 저장하기 위한 트리 형태의 자료구조 이다. 각 노드는 하나의 문자를 나타내며 문자열의 공통 접두사를 공유하는 구조를 가진다.  
문자열 검색은 문자열 길이를 L이라고 할떄 O(L) 시간에 수행할수있기 떄문에 자동완성이나 사전 검색과 같은 기능에서 많이 사용된다.  

KMP 알고리즘은 문자열 패턴 검색 알고리즘이다.  
패턴 내부의 접두사와 접미사가 같은 부분을 미리 계산하여 불필요한 비교를 줄이는 방식으로 동작한다.   
이 알고리즘은 문자열을 O(N+M)의 시간복잡도로 검색할수있다.  

Rabin-Karp 알고리즘은 해시 기반 문자열 검색 알고리즘이다.  
문자열을 해시값으로 변환하여 비교하는 방식으로 동작하며 평균적으로 O(N+M)의 시간복잡도를 가진다.

### 이진탐색이 무엇인지 설명하고, 시간복잡도를 증명해 보세요.

이진탐색은 정렬된 배열에서 특정 값을 찾기 위한 알고리즘이다.   
배열의 중간값을 선택하여 찾는 값과 비교하고, 그 결과에 따라 탐색 범위를 왼쪽 절반 또는 오른쪽 절반으로 줄인다.  
이과정을 반복해서 원하는 값을 찾는다. 

이진탐색에서는 탐색 범위가 매 단계마다 절반으로 줄어든다.  
데이터의 개수가 n일때 탐색 과정은 n, n/2, n/4, n/8과 같이 줄어든다.  
k번의 탐색후 탐색 범위가 1이 된다고 하면 n/2^k=1이 되고 이를 정리하면 2^k=n이 된다.  
따라서 k는 log₂ n이 되며 이진탐색의 시간복잡도는 O(log n)이 된다.

### Lower Bound, Upper Bound 는 무엇이고, 이를 어떻게 구현할 수 있을까요?
Lower Bound는 정렬된 배열에서 특정 값 이상이 처음 등장하는 위치를 찾는것이다.  
반면 Upper Bound는 특정값보다 큰 값이 처음 등장하는 위치를 찾는것이다.  
배열이 1,2,2,2,3,4라고할때 lower_bound(2)는 2가 처음 등장하는 위치인 index 1을 반환하고,  
upper_bound(2)는 값 2보다 큰 값이 처음 등장하는 위치인 index 4를 반환한다.  
이런 연산은 이진탐색을 기반으로 구현할수 있으며, mid값을 기준으로 탐색 범위를 줄여가면서 조건에 맞는 위치를 찾는다. 

### 이진탐색의 논리를 적용하여 삼진탐색을 작성한다고 가정한다면, 시간복잡도는 어떻게 변화할까요? (실제 존재하는 삼진탐색 알고리즘은 무시하세요!)
삼진탐색은 탐색 범위를 세 구간으로 나누는 방식이다.  
탐색 과정에서 배열을 세 부분으로 나누고 두개의 기준점을 사용하여 탐색 범위를 줄인다.  
이경우 탐색 범위는 n, n/3, n/9, n/27과 같이 줄어든다.  
k번 탐색후 탐색 범위가 1이 된다고 하면, n/3^k=1이 되고 이를 정리하면 3^k=n이된다.  
따라서 k는 log₃ n이 되며 시간복잡도는 O(log₃ n)이 된다.  
다만 실제 구현에서는 비교연산이 더 많아지기때문에 일반적으로는 이진 탐색이 더 효율적이다. 


### 기존 이진탐색 로직에서 부등호의 범위가 바뀐다면, 어떤 영향이 있을까요?
이진 탐색에서 사용하는 부등호 조건은 탐색 범위를 어떻게 줄일지 결정한다.  
따라서 부등호 조건이 달라지면 탐색 결과가 달라질수있다.  
< 조건을 사용할지 <= 조건을 사용할지에 따라 탐색 범위를 이동시키는 방식이 달라지며, 이는 lower bound와 upper bound와 같은 서로 다른 결과를 만들수있다.  
또한 조건이 잘못 설정되면 탐색 범위가 줄어들지 않아 무한 루프가발생하거나 경계값 오류가 발생핤수도있다. 

### 그리디 알고리즘과 동적 계획법을 비교해 주세요.
그리디 알고리즘은 매 단계에서 현재 시점에서 가장 최적이라고 판단되는 선택을 하는 방식이다.  
각 단계의 선택이 전체 최적해로 이어진다는 조건이 성립할때 사용할수있다.  
반면 동적 계획법은 문제를 여러 개의 작은 부분 문제로 나누어 해결하는 방식이다.  
이미 계산한 결과를 저장하여 같은 계산이 반복되지 않도록 하며, 이를 통해 전체 문제의 최적해를 구한다.  
그리디 알고리즘은 구현이 비교적 간단하고 계산 속도가 빠르지만, 항상 최적해를 보장하지는 않는다. 반면 동적 계획법은 계산 비용이 더 들지만 최적해를 보장할수있다. 

### 그렇다면, 어떤 경우에 각각의 기법을 사용할 수 있을까요?
그리디 알고리즘은 각 단계에서 최적 선택이 전체 최적해로 이어지는 경우에 사용할수있다.  
예를들어 활동선택 문제나 최소 스패닝 트리와 같은 문제에서 사용된다.  

동적계획법은 문제를 부분문제로 나눌수있고, 동일한 부분문제가 반복적으로 등장하는 경우에 사용할수있다.  
대표적인 예로는 피보나치 수열, 최장공통 부분수열, 배낭문제 등이 있다. 

### 동적 계획법으로 풀 수 있는 모든 문제는 재귀로 변환하여 풀 수 있나요?
가능하다. 동적 계획법은 기본적으로 재귀적인 점화식을 기반으로 하기때문에 모든 DP문제는 재귀방식으로 표현할수있다.  
이를 Top-down 방식이라고 하며 메모이제이션을 사용해 이미 계산한 결과를 저장한다.  
다만 재귀 방식은 함수 호출 스택을 사용하기 때문에 오버헤드가 발생하거나 stack overflow가 발생할수있다.  
그래서 실제 구현에서는 반복문 기반의 Bottom-up방식이 더 많이 사용된다.

# 고급 자료구조

## 면접대비 질문

### B-Tree와 B+ Tree의 차이점과 각각의 사용 사례를 설명해주세요.
B-Tree는 하나의 노드가 여러개의 키와 자식을 가질수있는 균형트리구조이다.  
데이터는 내부노드와 리프노드 모두에 저장될수있으며 탐색 과정에서 내부노드에서도 데이터를 찾을수있다.  

B+Tree는 내부노드에는 키 값만 저장하고 실제 데이터는 모두 리프노드에 저장하는 구조이다.  
또한 리프노드들이 연결 리스트 형태로 연결되어있기때문에 순차 탐색이 효율적이다.  

B-Tree는 파일 시스템과 같은 저장구조에서 사용되기도하지만, 범위 검색이 중요한 데이터베이스에서는 B+Tree기 인덱스 구조로 널리 사용된다. 

### 데이터베이스에서 B+ Tree를 인덱스 구조로 사용하는 이유는 무엇인가요?
B+Tree는 모든 실제 데이터가 리프노드에 저장되고 리프노드들이 연결 리스트 형태로 연결되기때문에 범위 검색에 매우 효율적이다.  
예를들어 특정 값 사이의 데이터를 조회하는 경우 리프노드를 순차적으로 탐색하면 된다.  
또한 하나의 노드에서 많은 키를 저장할수있기때문에 트리의 높이가 낮아지고, 그 결과 디스크 접근 횟수를 줄일수있다.  
이러한 이유로 대부분의 관계형 데이터베이스는 인덱스 구조로 B+Tree를 사용한다. 

### LRU 캐시를 O(1) 시간복잡도로 구현하는 방법을 설명해주세요.
LRU 캐시는 가장 오래 사용되지 않은 데이터를 제거하는 캐시 정책이다.  
이를 O(1) 시간복잡도로 구현하기 위해서는 HashMap과 Doubly Linked List를 함께 사용한다.  
해쉬맵은 키를 이용해 노드를 빠르게 찾기 위해 사용되고, 더블링크드리스트는 데이터의 사용 순서를 관리하기 위해 사용된다.  
데이터에 접근하면 해당 노드를 리스트의 앞쪽으로 이동시키고, 캐시 용량을 초과하면 리스트의 가장 뒤에 있는 노드를 제거한다.  
이 구조를 사용하면 조회와 삽입, 삭제 연산을 모두 O(1)에 수행할수있다. 

### LRU와 LFU 캐시 교체 정책의 차이점과 구현 방법을 설명해주세요.
LRU는 가장 오래 사용되지 않은 데이터를 제거하는 방식이다. 최근에 사용된 기록을 기준으로 캐시를 교체한다.  
LFU는 사용빈도가 가장 낮은 데이터를 제거하는 방식이다. 데이터가 얼마나 자주 사용되었는지를 기준으로 캐시를 교체한다.  
LRU는 해시맵과 더블링크드리스트를 이용해 간단하게 구현할수있다.  
LFU는 데이터의 접근횟수를 관리해야하기때문에 해시맵과 빈도 기반 리스트 등의 구조를 함께 사용해야하며 구현이 더 복잡하다. 

### 블룸 필터(Bloom Filter)의 원리와 사용 사례를 설명해주세요.
블룸필터는 확률 기반의 집합 자료구조로 특정 데이터가 집합에 존재하는지 빠르게 확인하기위해 사용된다.  
여러개의 해시 함수를 사용하여 데이터를 해시 값으로 변환하고, 해당 위치의 비트를 1로 설정한다.  
이후 데이터를 조회할떄 동일한 해시 함수를 적용하여 해당 위치의 비트들이 모두 1인지 확인한다.  
블룸필터는 메모리 사용량이 적고 조회속도가 빠르기 때문에 캐시 시스템, 데이터베이스 조회 최적화, CDN시스템 등에서 사용된다. 

### 블룸 필터의 False Positive와 False Negative 중 어떤 것이 발생할 수 있고, 그 이유는 무엇인가요?
블룸필터에서는 False Positive가 발생할수있지만, False Negative는 발생하지 않는다.  
false Positive는 실제로 존재하지 않는 데이터인데도 존재한다고 판단하는 경우이다. 여러개의 해시 함수가 같은 비트위치를 공유할수있기 때문에 이러한 현상이 발생한다.  
반면 false Negative는 발생하지 않는데, 블룸필터는 데이터를 추가할떄 비트를 0에서1로만 변경하며 다시 0으로 되돌리는 연산이 없기때문에 실제로 저장된 데이터가 존재하지 않는다고 판단되는 경우는 발생하지 않는다. 

### Skip List의 구조와 탐색 원리를 설명해주세요.
Skip List는 여러 레벨의 연결 리스트로 구성된 확률 기반 자료구조이다.  
가장 아래 레벨은 일반적인 정렬된 Linked List 구조이며, 위쪽 레벨로 갈수록 일부 노드만 포함되어 있어 더 빠르게 이동할 수 있도록 구성된다.  
탐색은 가장 높은 레벨에서 시작하여 오른쪽으로 이동하면서 다음 노드의 값이 목표 값보다 크면 한 단계 아래 레벨로 내려가는 방식으로 진행된다.  
이러한 과정을 반복하여 원하는 값을 찾는다.  
Skip List는 균형 트리처럼 복잡한 재구성 과정 없이도 평균적으로 O(log n)의 탐색 성능을 제공한다. Redis의 Sorted Set과 같은 시스템에서 활용되는 자료구조이다.
